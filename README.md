# Bachelor Thesis – Search Engine Case Study

Comparing ingest and search performance of **Typesense**, **Meilisearch** and
**Elasticsearch** on the same DBpedia dataset. Each engine runs in
Docker, is fed an identical JSONL extract, and is benchmarked with
[k6](https://k6.io/).

## Repository layout

```
.
├── generate_datasets.py       # corpus.jsonl -> fixed-size JSONL subsets
├── data/                      # Generated datasets (tracked via Git LFS)
├── typesense/                 # docker-compose for Typesense
├── meilisearch/               # docker-compose for Meilisearch
├── elasticsearch/             # docker-compose for Elasticsearch
├── monitoring/stats.py        # avg/max CPU+RAM per container (local or remote)
└── testing/k6/
    ├── docker-compose.yml     # k6 runner, one service per scenario
    ├── data/queries.jsonl     # DBpedia-Entity v2 query set (English)
    ├── results/               # k6 raw + summary output (gitignored)
    └── scripts/
        ├── config.js          # single source of truth for host, ports, …
        ├── <engine>-ingest.js              # bulk load + doc count check
        └── <engine>-search.js              # ramping RPS -> breaking point
```

`<engine>` is one of `typesense`, `meilisearch`, `elasticsearch`.

## Quick start

```bash
# 1. Pull datasets tracked via Git LFS
git lfs install
git lfs pull

# 2. Generate a dataset (subset of corpus.jsonl)
python generate_datasets.py 10000              # -> data/wiki_10000_documents.jsonl
python generate_datasets.py 10000 100000       # one file per requested size

# 3. Configure target host / dataset in testing/k6/scripts/config.js

# 4. Start the engine
docker compose -f typesense/docker-compose.yml up -d

# 5. Run benchmarks (same pattern for meilisearch and elasticsearch).
#    Wrap each run in stats.py to record container CPU/RAM for the
#    duration of that run, use compose command directly to not monitor ressources
python monitoring/stats.py --host ssh://user@engine-host -- \
  docker compose -f testing/k6/docker-compose.yml run --rm typesense-ingest

python monitoring/stats.py --host ssh://user@engine-host -- \
  docker compose -f testing/k6/docker-compose.yml run --rm typesense-search
```

Results are written to `testing/k6/results/` (raw JSON + summary).

## Measuring index sizes

Index sizes were measured after ingest by running `du -sh`, Typesense saves index in Ram, measured db size instead

```bash
docker compose -f typesense/docker-compose.yml exec typesense du -sh /data/db
docker compose -f meilisearch/docker-compose.yml exec meilisearch du -sh /meili_data/data.ms/indexes
docker compose -f elasticsearch/docker-compose.yml exec elasticsearch du -sh /usr/share/elasticsearch/data/indices
```

## Notes

- All engines are capped at 2 CPUs / 4 GB RAM for comparability.
- All engines ingest in identical 10000-doc batches (`DOCS_PER_BATCH` in
  `config.js`), so client behaviour is the same for every engine.
- Search tests walk the term list round-robin, so every run issues the same
  deterministic query mix.
- Datasets in `data/` are tracked via Git LFS.
